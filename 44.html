What's New

This is LDPL 4.4 'Friendly Falcarius'. Rawrrr! This release fixes all the bugs found in LDPL 4.3, introduces os-specific flags, increases the usability of multicontainers and marks a new milestone in the history of LDPL.

Changelog

This is a list of things that have changed in every release of this LDPL version:

Release 4.4.3 (Released Dec 21, 2019)

  • [Fix] Fixed nested multicontainer accesses that were not working the way they should.
  • [Fix] Fixed the modulo statement as it was losing precision when working with negative values.

Release 4.4.2 (Released Dec 18, 2019)

  • [Add] Added an (i386) note to the version information of the 32 bit release of LDPL so it reads LDPL 4.4.2 (i386) when compiled for x86 platforms.
  • [Chg] Moved the definitions of the methods of the LDPL data type classes from ldpl-types.h into the LDPL compiler.

Release 4.4.1 (Released Dec 13, 2019)

  • [Fix] The compiler wasn't identifying <number>:<index> as invalid code and tried to compile it, triggering C++ compiler errors.
  • [Chg] The LDPL man page has been updated to the latest version of the documentation.

Release 4.4 (Released Dec 2, 2019)

  • [Add] Added an alternative <container> of <type> syntax for multicontainer declaration.
  • [Fix] File loading now loads files without line breaks at the end correctly.
  • [Fix] Fixed sub-procedure parameters that weren't working when declared as multicontainers.
  • [Chg] The PUSH MAP statement now works with multi-dimensional lists.
  • [Add] The FOR EACH statement now can be used with multicontainers.
  • [Add] OS names can now be used with the FLAG statement to set OS specific flags.
  • [Fix] The USING PACKAGE statement wasn't working in 4.3 and has been fixed.

Known Bugs and Workarounds

This is a list of bugs that we have found in this release and possible workarounds for them.

  • Some nested multicontainer accesses are not working as they should. The compiler seems to be mistaking invalid accesses as valid and valid ones as invalid.
    (Found Dec 21, 2019 - Fixed in 4.4.3)
  • The modulo operator loses precision when working with negative values.
    (Found Dec 19, 2019 - Fixed in 4.4.3)
  • The way the compiler defines the methods of the data types used in LDPL in LDPL 4.4.2 may lead to undefined behaviour problems in future releases of the language, should those definitions be modified. While this is not a problem at the moment, the use of ldpl-types.h will be deprecated in future releases of the language in favour of a compiler-centered declaration of those types.
    (Found Dec 18, 2019 - Fixed in 4.4.2)
  • The compiler isn't identifying <number>:<index> as invalid code (it is, numbers don't have subindexes) and tries to compile it, triggering C++ compiler errors. Workaround: do not write <number>:<index>, as it is invalid code.
    (Found Dec 5, 2019 - Fixed in 4.4.1)

Platform Support

LDPL 4.4 has been tested on Linux amd64, Linux i386 and macOS 64-bits.

It should also run without any problems on i386 and amd64 BSD systems, 32 macOS, mac OS X, ARMv8 Linux, Android Termux, PPC Linux and PPC OS X. If you are on one of these platforms, please download the source code and compile LDPL yourself by following the instructions detailed on README.md. In order to build and use the LDPL compiler, a C++11 compiler is required on your $PATH.

How to Install

If you are downloaded a pre-compiled binary, just copy it to a folder on your PATH (for example /bin or /usr/local/bin) and you are ready to go. If the LPM package manager was included in the downloaded package, you may copy it to that same location or any other folder on your PATH. For more information on what folders are on your PATH run the command echo $PATH.

For more information and alternate ways to install LDPL including building it yourself, please read this.
